# Active Listening Skill

曖昧な要求から真のニーズとビジネス価値を引き出します。

## Input
- 曖昧な要求や仕様
- ステークホルダーの口頭説明
- 不完全なユーザーストーリー
- 漠然としたアイデア

## Output
- 構造化された要求理解
- 明確化されたユーザーペルソナ
- 特定されたビジネス価値
- 確定したスコープ境界
- 初期仮説リスト

## Core Workflow

Active Listeningは線形的なフェーズではなく、必要に応じて以下を行き来する対話プロセスです:

### 1. 初期理解 (5分)

要求の大枠を把握

```
Claude: 「要求について教えてください。
        大まかな説明から始めていただいて構いません。」

→ オープンクエスチョンで自由に話してもらう
```

### 2. 深掘り (10-15分)

**5 Whys** で根本原因を探る:
```
「なぜこの機能が必要なのですか?」
→ 3-5回繰り返して真の目的を発見
```

**ソクラテス式問答** で仮定を明らかにする:
- 明確化: 「具体的にどういう意味ですか?」
- 仮定の探求: 「なぜ〜だけでは不十分なのですか?」
- 証拠の要求: 「データはありますか?」
- 視点の転換: 「他の解決方法は?」
- 含意の検討: 「〜すると、〜になりますが?」

**5W1H** で視野を広げる:
- What: 「何を実現したいですか?」
- Who: 「誰が使いますか?」
- When: 「いつ使いますか?」
- Where: 「どこで使いますか?」
- Why: 「なぜ必要ですか?」
- How: 「どのように動作すべきですか?」

### 3. ペルソナ・コンテキスト明確化 (5分)

```
「この機能を使う典型的なユーザーについて教えてください:
 - どんな人ですか? (年齢、職業、ITリテラシー)
 - どんな状況で使いますか?
 - どんな目的で使いますか?」
```

### 4. スコープ境界確定 (5分)

```
「この機能に含めるもの、含めないものを明確にしましょう:

 【含める】✅
 [リストアップ]

 【含めない】❌
 [リストアップ]

 【保留・要検討】❓
 [リストアップ]

 これで合っていますか?」
```

### 5. 仮説提示と検証 (5分)

```
「現時点での私の理解をまとめます:

 【目的】
 ...

 【ペルソナ】
 ...

 【スコープ】
 ...

 この理解で正しいですか?
 修正や追加はありますか?」
```

## Core Principles

### 1. 判断を保留する

```
❌ Bad:
ユーザー: 「検索機能が欲しい」
Claude: 「Elasticsearchで実装しますね」→ 技術に飛びつく

✅ Good:
ユーザー: 「検索機能が欲しい」
Claude: 「なぜ検索が必要ですか?」→ 問題を理解する
```

### 2. オープンクエスチョンを使う

```
❌ Closed: 「10件表示すれば良いですか?」→ Yes/No
✅ Open: 「どのように表示すべきですか?」→ 自由な回答
```

### 3. 具体例で確認する

```
「『ユーザーフレンドリー』とは?
 例: すぐ表示? 候補表示? シンプル画面?」
```

### 4. パラフレーズで検証

```
「つまり、不正アクセス防止のため、
 一定回数失敗したらロックする、
 という理解で合っていますか?」
```

## Output Format

`reference/output-template.md` の構造で成果物を作成:

```markdown
# Active Listening 成果物

## 要求の本質
- ビジネス価値
- 解決したい課題
- 成功指標 (KPI)

## ユーザーペルソナ
- プライマリーペルソナ
- セカンダリーペルソナ (該当時)

## スコープ
- 含めるもの (In Scope) ✅
- 含めないもの (Out of Scope) ❌
- 要検討 (TBD) ❓

## 前提条件と制約
- ビジネス制約
- 技術制約
- 時間制約

## 未解決の質問
- 🔴 Blocker (実装前に解決必須)
- 🟡 Clarification (品質向上)
- 🟢 Future (将来検討)

## 次のステップ
Example Mappingフェーズへ進む
```

## Do's & Don'ts

### DO ✅
1. 判断を保留: すぐに解決策を提案しない
2. オープンクエスチョン: Yes/No以外の質問
3. 具体例で確認: 抽象的表現を具体化
4. パラフレーズ: 理解を確認
5. 沈黙を活用: 考える時間を与える

### DON'T ❌
1. 技術的解決策に飛びつかない
2. 誘導尋問しない
3. 一度に多く聞かない
4. 仮定を放置しない
5. 反論しない

## Time Budget

- 通常20-30分以内で完了
- それ以上かかる場合:
  - 要求が複雑すぎる → ストーリー分割
  - 未解決質問が多い → 追加セッション
- Example Mapping前に必ず仮説検証完了
