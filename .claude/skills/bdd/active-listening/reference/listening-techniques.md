# Active Listening Techniques

## 技法1: ソクラテス式問答

仮定を明らかにし、論理的に深掘りする技法。

### パターン
1. 明確化の質問
2. 仮定の探求
3. 証拠の要求
4. 視点の転換
5. 含意の検討

### 実例: 検索機能の深掘り

```
ユーザー: 「検索機能が必要です」

[明確化]
Claude: 「検索機能が必要な理由を教えてください」
ユーザー: 「商品が多すぎて、目的の商品が見つからないから」

[仮定の探求]
Claude: 「『多すぎる』とは、具体的に何商品くらいですか?
        また、なぜカテゴリ分けだけでは不十分なのでしょうか?」
ユーザー: 「500商品以上あって、カテゴリが大雑把すぎる」

[証拠の要求]
Claude: 「ユーザーが商品を見つけられないというデータはありますか?
        例えば、カスタマーサポートへの問い合わせや、
        直帰率の高さなど」
ユーザー: 「はい、カスタマーサポートに週10件以上『商品が見つからない』という問い合わせがあります」

[視点の転換]
Claude: 「ユーザーの立場で考えてみましょう。
        検索以外に、商品を見つけやすくする方法はありますか?
        例えば、カテゴリの細分化、おすすめ商品の表示、
        フィルター機能など」
ユーザー: 「うーん、カテゴリの見直しも必要かもしれません」

[含意の検討]
Claude: 「検索機能を導入すると、
        ユーザーは商品名を正確に知っている必要がありますが、
        新規ユーザーは商品名を知らないかもしれません。
        この点はどうお考えですか?」
ユーザー: 「確かに。タグやカテゴリでも検索できるようにすべきですね」
```

## 技法2: 5 Whys

根本原因を発見する技法。

### 実例: ログイン機能の真の目的

```
ユーザー: 「ログイン機能が必要です」

Why 1: なぜログイン機能が必要なのですか?
→ 「ユーザーごとに異なる情報を表示したいから」

Why 2: なぜユーザーごとに異なる情報を表示したいのですか?
→ 「ユーザーの購入履歴に基づいたおすすめ商品を表示したいから」

Why 3: なぜ購入履歴に基づいたおすすめ商品を表示したいのですか?
→ 「リピート購入率を上げたいから」

Why 4: なぜリピート購入率を上げたいのですか?
→ 「新規顧客獲得コストが高いから、既存顧客からの売上を最大化したい」

Why 5: なぜ既存顧客からの売上を最大化したいのですか?
→ 「今期の売上目標を達成するため」

Claude: 「理解しました。真の目的は『リピート購入率向上による売上最大化』ですね。
        ログイン機能は手段の一つであり、
        ユーザー識別とパーソナライゼーションが本質的な要求ですね。」
```

## 技法3: 仮説検証

自分の理解を仮説として提示し、検証する技法。

### 実例

```
Claude: 「現時点での私の理解は以下です。正しいですか?

        【仮説1】 機能の目的
        - ユーザーが商品を素早く見つけられるようにする
        - 目標: カスタマーサポートへの問い合わせを50%削減

        【仮説2】 ユーザーペルソナ
        - 既に購入経験のあるリピーター
        - 商品名やカテゴリを大まかに知っている
        - モバイルからのアクセスが多い

        【仮説3】 検索の範囲
        - 商品名、商品説明、タグを対象
        - カテゴリフィルターとの併用可能
        - 価格帯や在庫状況での絞り込みは対象外

        この仮説で抜けている重要な点はありますか?」

ユーザー: 「大体合っています。ただ、新規ユーザーも対象です。
          あと、価格帯フィルターは欲しいです。」

Claude: 「ありがとうございます! 仮説を修正します。
        新規ユーザーの場合、商品名を知らないケースが多いと思いますが、
        どのように検索してもらうことを想定していますか?」
```

## 技法4: オープンクエスチョン 5W1H

### What (何を)
- 「具体的に何を実現したいのですか?」
- 「この機能で何が変わりますか?」

### Who (誰が)
- 「この機能を使うのは誰ですか?」
- 「ステークホルダーは誰ですか?」

### When (いつ)
- 「ユーザーはどんな時にこの機能を使いますか?」
- 「いつまでに必要ですか?」

### Where (どこで)
- 「どの画面・どのタイミングで使いますか?」
- 「どのプラットフォーム(Web/モバイル)で必要ですか?」

### Why (なぜ)
- 「なぜこの機能が必要なのですか?」
- 「なぜ今なのですか?」

### How (どのように)
- 「理想的には、どのように動作すべきですか?」
- 「成功をどのように測定しますか?」

## Core Principles

### 1. 判断を保留する

```
❌ Bad:
ユーザー: 「検索機能が欲しい」
Claude: 「分かりました。Elasticsearchを使って全文検索を実装しますね。」
→ 技術的解決策に飛びつき、真のニーズを理解していない

✅ Good:
ユーザー: 「検索機能が欲しい」
Claude: 「検索機能が必要な理由を教えていただけますか?
        現在、ユーザーはどのように商品を見つけていますか?」
→ 技術ではなく、問題の本質を理解しようとしている
```

### 2. オープンクエスチョンを使う

```
❌ Closed Question:
「検索結果は10件表示すれば良いですか?」
→ Yes/Noで答えられる、選択肢を限定している

✅ Open Question:
「検索結果はどのように表示すべきでしょうか?」
→ 相手の考えを自由に引き出せる
```

### 3. 具体例で確認する

```
Claude: 「『ユーザーフレンドリー』とおっしゃいましたが、
        具体的にはどんな体験を想像されていますか?

        例えば:
        - ボタンを押したらすぐに結果が表示される?
        - 入力途中で候補が表示される?
        - シンプルな画面構成?

        どれが近いですか? あるいは別のイメージですか?」
```

### 4. パラフレーズで理解を確認する

```
ユーザー: 「ログイン失敗が続いたら、アカウントをロックしたい」

Claude: 「理解を確認させてください。
        つまり、不正アクセスを防ぐために、
        一定回数ログインに失敗したユーザーは、
        一時的にログインできなくする、という理解で合っていますか?」

ユーザー: 「はい、その通りです」
```

## Do's & Don'ts

### DO ✅
1. 判断を保留する: すぐに解決策を提案しない
2. オープンクエスチョンを使う: Yes/Noで答えられない質問
3. 具体例で確認する: 抽象的な表現を具体化
4. パラフレーズする: 理解を相手の言葉で確認
5. 沈黙を活用する: 相手に考える時間を与える

### DON'T ❌
1. 技術的解決策に飛びつかない: まずは問題を理解
2. 誘導尋問をしない: 自分の仮説を押し付けない
3. 一度に多く聞かない: 1つずつ丁寧に
4. 仮定を放置しない: 曖昧な言葉は必ず確認
5. 反論しない: まずは相手の意見を受け止める
